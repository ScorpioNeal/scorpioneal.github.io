Android 关于onPause和onStop回调时机的问题

Posted on 2019-07-02 | In Android | |

问题

我们知道一个Activity的启动会执行onCreate -> onStart -> onResume

一个Activity的停止会执行 onPause -> onStop -> onDestroy

那么，ActivityA启动ActivityB的时候，ActivityB的onCreate与ActivityA的onPause还是onStop哪个在前呢?

区别

先看下onPause和onStop官方文档的解释

  • onPause

    Called when the activity loses foreground state, is no longer focusable or before transition to stopped/hidden or destroyed state. The activity is still visible to user, so it’s recommended to keep it visually active and continue updating the UI. Implementations of this method must be very quick because the next activity will not be resumed until this method returns. 也就是说这个onPause方法里不能执行太复杂的方法，因为另一个Activity的onResume要等这个Activity的onPause执行完才会执行，也就是如果这里做了大量的操作，启动下一个Activity就会被卡住

  • onStop

    Called when the activity is no longer visible to the user. This may happen either because a new activity is being started on top, an existing one is being brought in front of this one, or this one is being destroyed. This is typically used to stop animations and refreshing the UI, etc. 常在这里来停止动画和停止刷新UI.

验证

代码很简单.

结论是

ActivityA onPause -> ActivityB onCreate -> ActivityB onStart -> ActivityB onResume -> ActivityA onStop

结论

其实从官方文档就可以看出, ActivityB会在ActivityA的onPause执行之后才会执行，所以文档里建议不要再onPause里做耗时操作，不然会影响Activity启动速度. 那为什么onStop会在ActivityA执行完create, start, resume之后才会执行呢？也是为了让画面更流畅，用户可以点击跳转后可以快速的切换到下一个Activity，体验更流畅些，在ActivityA看不到的时候，才去做一些A中的onStop里执行的方法.
